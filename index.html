Foundational ideas
01 · Policy

A policy is the robot's rule for choosing the next move.

Write a policy as $\pi(a \mid o, c)$: choose an action $a$ from an observation $o$ and optional context $c$. The observation might be camera pixels and joint angles. The context might be a language instruction, robot type, or visual goal.

In the toy arm, you are the policy. You see the target and choose two joint angles. A learned robot policy does the same kind of mapping, but in a much larger action space.

The rest of the guide asks one question repeatedly: what information should the policy condition on before it acts?

02 · Actions and chunks

Robots usually need short behaviors, not isolated twitches.

An action can be a single motor command, but manipulation often works better when the model emits a short chunk: a few future moves that belong together.

Move the chunk length. Single-step control reacts every frame and can jitter. Longer chunks commit to a local plan, which is why modern robot policies often predict continuous action chunks.

π0.7's action expert outputs continuous chunks, so it can express a small behavior like reach, grasp, lift, or press.

Part 1 · learning from demonstrations
03 · Behavior cloning

Imitation learning copies the action a demonstrator took.

Behavior cloning records demonstrations and trains a model to predict the demonstrated action from the observation. It is the simplest bridge from data to policy.

Click to add demos. The fitted curve is confident near examples and shaky away from them. In robotics, "near" means similar camera views, object poses, robot states, and task context.

Imitation is powerful where the demonstrations cover the situation. It becomes brittle when the robot drifts into states no one demonstrated.

04 · Covariate shift

Small imitation errors can move the robot off the demo path.

A policy trained only on expert states may not know what to do after its own mistake. One slight miss changes the next observation, which can cause the next action to be worse.

Step the rollout. The yellow band is the demonstration corridor. The learned policy is fine inside it, but becomes less reliable as the state drifts away.

This is why broad data, recovery data, and good context matter: the robot has to act in the world it creates, not only the world in the demonstrations.

05 · Multimodality

One observation can have several correct futures.

Human demonstrators do not move identically. One reaches high, another curls around an obstacle, and both can be right. If a model averages those choices, the result can be a motion no demonstrator actually used.

The toy arm shows three valid trajectories to the same target. The correct lesson is not "pick the average"; it is "represent the choices."

Robot action data is multimodal. Good policies need to sample a plausible behavior mode, not collapse all modes into one mean action.

Part 2 · generative action models
06 · Flow matching

A generative policy turns noise into an action trajectory.

Instead of regressing to one averaged action, a flow-matching policy learns a vector field over action space. At inference time, it starts from noise and follows the field toward a plausible action chunk.

The visualization is two-dimensional. In a robot model, the points are short continuous action sequences, conditioned on the observation and context.

Flow matching gives the policy a way to keep multiple behavior modes separate while still producing smooth continuous actions.

07 · Vision-language-action

A VLA grounds semantic knowledge into robot motion.

A vision-language model knows visual and semantic concepts: drawers, handles, shirts, buttons. A vision-language-action model adds the ability to output robot actions.

The important bridge is grounding. The model cannot just say "open the drawer"; it has to convert that phrase and image into continuous motion.

π0.7 can be read as a VLA backbone plus an action expert, trained so language, images, and context steer the same policy.

Part 3 · context as the missing variable
08 · Naive mixing

Broad robot data creates contradictions unless the model knows the source.

The obvious recipe is to mix demonstrations from every robot and train one policy. The problem is that similar observations can require different actions because robot bodies, grippers, control interfaces, and strategies differ.

Turn on the context label. The same data stops looking contradictory because the model is told which behavior mode it should imitate.

A generalist needs broad data, but it also needs context that explains how each behavior should be interpreted.

09 · Context coordinates

Context locates the behavior you want inside a large policy.

Think of context as coordinates. Language says the semantic goal. Embodiment or control metadata says which robot and interface. Quality and speed tags say which style of behavior to prefer. Visual subgoals say where the scene should go next.

PI's phrase for π0.7 is diverse context. The point is not extra decoration around the prompt; it is the variable that makes broad robot data trainable and the final policy steerable.

A steerable model is a conditional model: change the coordinates, change the behavior without changing the checkpoint.

10 · Visual subgoals

A world model can turn a task into a target image.

Some spatial instructions are easier to show than to say. A lightweight world model can predict what the scene should look like next, and the action policy can act toward that target image.

This separates high-level intent from low-level control: first imagine the desired near-future pixels, then choose the joint motion that moves the real scene there.

Visual subgoals are context too. They steer the policy with pixels instead of only words.

Part 5 · π0.7 as case studies
12 · Compositional task

The air fryer is a test of context-guided recombination.

The model has related primitives: open hinged objects, grasp handles, place food, press buttons. The unfamiliar part is arranging those primitives for this appliance.

Move through the tabs. Plain language gets partway there. More explicit substeps and visual subgoals make the same policy easier to steer.

The concept to notice is not magic novelty. It is conditional composition: known skills arranged by context.

13 · Embodiment

Transfer means mapping a task onto a different body.

The UR5e example is not just language following. The robot body changes: size, inertia, gripper geometry, and teleoperation difficulty all shift.

That is why embodiment is a context variable. The policy has to solve the same task through a different physical interface.

Cross-embodiment transfer is the robotics version of analogy: same goal, different body, adapted strategy.

15 · Limits

Foundation does not mean solved.

Unseen tasks remain harder than familiar task families, and broad datasets make it difficult to prove that a behavior is entirely novel rather than a new combination of related experience.

The next frontier is a policy that can plan, act, observe, and revise over long horizons with the same reliability that it executes short chunks.

π0.7 is best read as a strong step toward steerable robot foundation models, not as the end of robotics.

Open problems
Novel tasksZero-shot and compositional tasks are still less reliable than familiar task families.
Novelty auditWith large robot datasets, deciding whether a behavior is truly unseen is technically difficult.
Reflective planningFuture systems need to reason through plans, see outcomes, and revise before acting again.
